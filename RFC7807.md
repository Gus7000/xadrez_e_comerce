# RFC 7807 no projeto

Este projeto utiliza o formato Problem Details for HTTP APIs (RFC 7807) para respostas de erro.

## Content-Type

As respostas de erro retornam:

- `application/problem+json`

## Estrutura do payload

Os campos retornados sao:

- `type`: URI que identifica o tipo do problema.
- `title`: titulo curto e legivel do erro.
- `status`: codigo HTTP.
- `detail`: descricao detalhada do erro.
- `instance`: caminho da requisicao que gerou o erro.
- `timestamp`: data/hora da geracao do erro.
- `errors` (opcional): lista de erros de campo para validacao.

## Exemplo de erro de validacao (422)

Esse caso e retornado por `ConstraintViolationExceptionMapper` para falhas de Bean Validation.

```json
{
  "type": "https://example.com/problems/validation-error",
  "title": "Validation Error",
  "status": 422,
  "detail": "Um ou mais campos nao passaram na validacao.",
  "instance": "/material",
  "timestamp": "2026-04-22T16:10:45.123",
  "errors": [
    {
      "field": "nome",
      "message": "nao pode estar em branco"
    }
  ]
}
```

## Exemplo de recurso nao encontrado (404)

Esse caso e retornado por `NotFoundExceptionMapper`.

```json
{
  "type": "https://example.com/problems/not-found",
  "title": "Not Found",
  "status": 404,
  "detail": "Material nao encontrado",
  "instance": "/peca/1",
  "timestamp": "2026-04-22T16:11:02.456"
}
```

## Implementacao no codigo

- Modelo do erro: `ProblemDetails`
- Mapeadores:
  - `ConstraintViolationExceptionMapper`
  - `NotFoundExceptionMapper`
- Excecoes tratadas:
  - `jakarta.validation.ConstraintViolationException`
  - `jakarta.ws.rs.NotFoundException`

## Quando este formato e usado

- Erros de validacao em payloads de entrada com Bean Validation: retorna 422.
- Recursos inexistentes ou enumeracoes invalidas mapeadas para `NotFoundException`: retorna 404.
