# RFC 7807 no projeto

Este projeto utiliza o formato **Problem Details for HTTP APIs** (RFC 7807) para respostas de erro de validacao.

## Content-Type

As respostas de erro retornam:

- `application/problem+json`

## Estrutura do payload

Os campos retornados sao:

- `type`: URI que identifica o tipo do problema.
- `title`: titulo curto e legivel do erro.
- `status`: codigo HTTP.
- `detail`: descricao detalhada (inclui mensagens de validacao).
- `instance`: caminho da requisicao que gerou o erro.

## Exemplo de erro de validacao (400)

```json
{
  "type": "https://example.com/problems/validation-error",
  "title": "Validation Error",
  "status": 400,
  "detail": "nome: nome nao pode estar em branco",
  "instance": "/material"
}
```

## Exemplo de recurso nao encontrado (404)

```json
{
  "type": "https://example.com/problems/not-found",
  "title": "Not Found",
  "status": 404,
  "detail": "Material nûÈo encontrado",
  "instance": "/peca/1"
}
```

## Implementacao no codigo

- Modelo do erro: `ProblemDetails`
- Mapper: `ConstraintViolationExceptionMapper`
- Excecoes tratadas:
  - `jakarta.validation.ConstraintViolationException`
  - `jakarta.ws.rs.NotFoundException`

## Quando este formato e usado

Atualmente, ele e usado para erros de validacao de Bean Validation em endpoints REST (por exemplo, `POST` e `PUT` com `@Valid`).
